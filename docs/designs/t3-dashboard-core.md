## Vision

### 10x Check
T3 isn't "add 4 pages to the dashboard." It's transforming PixelPort from a redirect button into a living command center. Agent cards pulse when working. The Home page tells you what needs attention right now. Approvals are inline-editable — approve content in 90 seconds without leaving the page. Every view has skeleton loading that matches content shape. The sidebar badges keep you informed at a glance. The dashboard feels like watching someone work, not reading a report.

### Platonic Ideal
You open PixelPort. The Home page immediately says: "3 posts need your approval. Your Chief of Staff is currently analyzing competitor X's latest tweet. Cost this week: $2.40." Agent cards have a subtle green pulse when online — the Chief is working. You click the approval badge, swipe through 3 content cards, approve 2, edit 1 inline, done in 90 seconds. The Kanban board shows 12 tasks organized. Run history: morning scan completed in 47 seconds, $0.12. Everything feels immediate, warm, intentional. You close the tab feeling informed and in control.

## Architecture

Implementation approach: **Shared Data Layer + View Pages**
- Centralized proxy hooks (`usePaperclip*`) using TanStack Query with shared cache
- **Pattern:** TanStack Query hooks per resource type (not React Context or Zustand). Cross-view data sharing via TanStack Query's built-in cache. No custom state management.
- Individual page components consume shared hooks
- **Polling is client-side only** (browser polls Vercel proxy → proxy forwards to Paperclip). No server-side polling.
- Background polling intervals (fixed values):
  - Agent status: **15s** (`refetchInterval: 15000`)
  - Sidebar badges: **30s** (`refetchInterval: 30000`)
  - Run events (when viewing run detail): **5s**
  - All other data: **no auto-polling** (manual refetch on user action)
- **Cache invalidation:** After mutations (approve/reject/edit/status change), invalidate the relevant query key immediately + show optimistic UI via toast. Badge count updates on next poll cycle (30s max).
- **Error boundary:** One `<DashboardErrorBoundary>` wrapping `<Outlet />` in `Dashboard.tsx`. Shows "Something went wrong" card with retry button. Does not wrap individual views — each view handles its own loading/error states via TanStack Query.

## Color Resolution

**DESIGN.md says:** "Agent pulse: Subtle green glow pulse on online agent dots" + Semantic green (#22C55E) = Success/Online.
**Full Wedge says:** "amber pulse on status dot" for online agents.

**Resolution:** Follow DESIGN.md semantics. Green pulse = online/healthy. Amber = actively running a task (not idle). Red = offline/error. This preserves semantic color meaning: green = good, amber = attention (something in progress), red = problem. Update Full Wedge description to match.

## Agent Activity Timeline Data Source

"Mini activity timeline (last 3 actions)" on agent cards:
- **Source:** `GET /companies/:id/heartbeat-runs?agentId={agentId}&limit=3` — returns the 3 most recent runs for that agent
- **Display:** Each entry shows: run result (success/fail icon), duration, relative timestamp
- **Fallback:** If no runs exist, show "No activity yet"

## Reviewer Concerns (Round 2 — 7 items)

### Fixed in plan:
Items below are implementation guidance to be followed during T3 build.

### RC-1: Inline Edit Flow (two-call sequence)
1. User clicks "Edit" → textarea in-place
2. User edits text, clicks "Approve" or "Save Draft"
3. If "Save Draft": `POST /approvals/:id/request-revision` with `{ comment: "User edited content", revisedContent: "..." }`
4. If "Approve": call `request-revision` → then `POST /approvals/:id/resubmit` → then `POST /approvals/:id/approve`
**BLOCKER CHECK:** Before implementing, verify `request-revision` body schema against Paperclip source at `v2026.318.0`. If it only accepts a comment/reason and not revised content, escalate to CTO — the inline edit flow must be redesigned.

### RC-2: Approval Queue Pagination
Use `limit=20, offset=0` for initial load. Defer infinite scroll/pagination to post-T3. Show count from sidebar-badges, not from list length.

### RC-3: Intelligence Brief Placeholder
Render a static card with label "Intelligence Brief" and body "Your Chief is analyzing — check back soon." No polling, no data fetch. Pure static UI.

### RC-4: Routing Auth
All new routes sit inside the existing `<ProtectedRoute>` wrapper in App.tsx. No new auth logic required.

### RC-5: Home Page Count Consistency
Home page pending approval count should derive from the same TanStack Query key as sidebar badges (`/sidebar-badges`), not from `/dashboard` summary. This ensures a single cache entry. Agent status on Home page uses the same `usePaperclipAgents` hook (15s) as Agent Cards view.

### RC-6: Optimistic Approval Removal
After approve/reject: (1) use `queryClient.setQueryData` to immediately remove the item from the pending list cache, (2) show success toast via Sonner, (3) `invalidateQueries` triggers background refetch. On API error: revert optimistic state and show error toast.

### RC-7: Quick Actions on Home Page
Two CTA buttons: "Review Approvals" (links to /dashboard/approvals) and "View Tasks" (links to /dashboard/tasks). Navigation shortcuts only — no backend calls.

### RC-8: Task Detail (Expand-to-Detail)
Shows issue title, description, status, current assignee, comment thread (`GET /issues/:id/comments`). Does not show heartbeat-context or active-run in T3 — deferred to T4.

### RC-9: Skeleton Shimmer Clarification
Skeleton shimmer uses amber at very low opacity (`bg-amber-500/5` or similar) to match DESIGN.md spec. At this opacity it doesn't carry semantic meaning — it's ambient warmth, not a warning signal.

## Engineering Decisions (from /plan-eng-review)

### ED-1: Inline Edit Flow (VERIFIED — 3-call sequence)
Paperclip `request-revision` only accepts `{ decisionNote, decidedByUserId }` — NO `revisedContent` field.
Revised content goes through `resubmit` which accepts `{ payload: Record<string, unknown> }`.

**Corrected flow:**
1. User clicks "Edit" → textarea in-place
2. "Save Draft": `POST /approvals/:id/request-revision` with `{ decisionNote: "User edited" }` → `POST /approvals/:id/resubmit` with `{ payload: { ...original, content: "edited" } }`
3. "Approve": same as Save Draft, then `POST /approvals/:id/approve`

### ED-2: Approval Queue is Type-Agnostic
Queue renders any approval regardless of type. No filtering by `hire_agent` vs `approve_ceo_strategy` vs future content types. When Chief agent starts creating content approvals, they appear automatically.

### ED-3: Shared `paperclipFetch()` Helper
All `usePaperclip*` hooks share a `paperclipFetch(path, options, token)` helper that handles:
- URL construction (`/api/tenant-proxy/` prefix)
- Authorization header (Bearer token from Supabase session)
- Response parsing + error classification
- Console.error logging on failure

### ED-4: @dnd-kit for Kanban Drag-and-Drop
Use `@dnd-kit/core` + `@dnd-kit/sortable` for task board. Modern, ~15kb, well-maintained.

### ED-5: Full Drag-and-Drop Kanban (not click-to-change)
CEO review decided: full DnD, not dropdown status change. Drag card between columns triggers `PATCH /issues/:id` with new status. Revert position on API error.

### ED-6: `beforeunload` Warning on Dirty Edit
When approval edit textarea is dirty, register `beforeunload` event. Browser shows native "Leave page?" dialog.

### ED-7: Plain Text Only for Approval Content (T3)
Approval content rendered as plain text — no link parsing, markdown, or HTML. URLs display as text strings. Rich rendering added as follow-up (with sanitization library). **UPDATE: Rich content rendering promoted to build-now in eng review.**

### ED-8: Console.error Logging on Proxy Failures
Each `usePaperclip*` hook logs `console.error` with endpoint path, status, and error message on failure.

### ED-9: 10 Test Suites Required
All new codepaths get test coverage:
1. `paperclipFetch()` helper (unit)
2. `usePaperclipAgents` hook (unit)
3. `usePaperclipApprovals` hook (unit)
4. `ProxyQueryWrapper` component (unit)
5. Approval approve mutation (integration)
6. Approval 3-call edit flow (integration)
7. Kanban drag-and-drop → PATCH (integration)
8. Cost coloring logic (unit)
9. Sidebar badge rendering (unit)

### DD-9: Sidebar Badge Animation
On count change: brief scale pulse (1.0 → 1.2 → 1.0, 300ms). Subtle peripheral attention cue.

### DD-10: Anti-Slop Implementation Guidance

- Run history: subtle card rows (not flat table), cost color coding, Geist Mono for duration
- Home page: approval count as prominent amber card at top, not a small number in a widget grid
- All cards get grain texture background (inherited from body::before at 3% opacity)

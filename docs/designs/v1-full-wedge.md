## Vision

### 10x Check
PixelPort isn't a dashboard for managing AI agents — it's an autonomous marketing engine that gets smarter every week. The Chief of Staff observes approval/rejection patterns, tracks post-publish performance, correlates content quality with engagement, and auto-adjusts its content strategy. Over 4-8 weeks, the approval rate trends upward and human editing burden trends downward. The agent becomes irreplaceable because it has learned *your* taste, *your* voice, *your* audience.

### Platonic Ideal
Monday morning. Open PixelPort. The Chief of Staff has already: monitored competitors over the weekend, drafted 12 social posts based on trending industry topics, scheduled 5 for the week at optimal engagement times (learned from prior posts), prepared a one-page intelligence brief. You spend 8 minutes reviewing. Approve 10, tweak 2, reject 0. The Chief of Staff notes your edits and adjusts. By week 4, you spend 3 minutes. By week 8, 95% approval rate without edits. Your competitors post 2x/week manually. You post daily with AI-calibrated content.

## Architecture Decisions

### Path 3: Hybrid (Custom Dashboard + Paperclip Headless + Admin View)
- **Custom React dashboard** on Vercel is the primary product experience (all team-facing UX)
- **Paperclip** runs headless as orchestration backend (tasks, runs, approvals, workspace files)
- **Paperclip UI** stays accessible as power-user/admin view via handoff token
- **Gateway token proxy** connects dashboard to Paperclip API (existing `proxyToGateway()` pattern)
- **Per-tenant isolation**: one DigitalOcean droplet per customer

### Data Source Split
| Data | Source | Why |
|---|---|---|
| Tasks / Kanban | Paperclip API (via proxy) | Paperclip owns task lifecycle |
| Agent status | Paperclip API (via proxy) | Agent runs on droplet |
| Run history | Paperclip API (via proxy) | Runs execute on droplet |
| Approvals | Paperclip API (via proxy) | Approval gates are Paperclip features |
| Workspace files (SOUL, vault) | Paperclip API (via proxy) | Files on droplet filesystem |
| Chat messages | Paperclip/OpenClaw (REST polling V1) | Real-time agent comms |
| Content performance metrics | Supabase | Fetched from X/LinkedIn APIs centrally |
| Competitor data | Supabase | Scraped centrally, shared across views |
| Integration status | Supabase | Integration registry already in Supabase |
| Team activity | Supabase | Cross-tenant audit log |
| Brand voice rules | Paperclip workspace vault | Stored as workspace files |
| User preferences | Supabase | Centralized user settings |

### Publishing Flow
**Approve = Schedule** — approval moves the post to the content calendar at the AI-suggested time. User can see it scheduled and cancel before it goes live.

**Publishing orchestration:** Inngest handles scheduled execution (fires at the scheduled time), retry logic (if platform API is down), and observability (track what published/failed). The unified social media API handles the actual platform posting.

**Idempotency:** All publish actions store `external_post_id` after successful publish. Before retry, check if post already exists to prevent double-posting.

### Content Data Ownership
Content pipeline data (drafts, approved posts, scheduled posts, published posts, media) lives in **Supabase** — not Paperclip. Paperclip manages task orchestration; Supabase owns the content lifecycle. This avoids dual-write complexity and makes the unified social API integration simpler.

### Team Roles (V1)
All authenticated users are admins. No role-based permissions for V1 — RBAC deferred to Phase 2 (see NOT in Scope).

### Media Support
Content pipeline includes AI-generated **images** (not just text). Approval queue uses a "media slot" pattern — each post has text + media slots that can be: generating, ready, failed, or user-uploaded. **Video deferred to V1.1** — requires CDN, transcoding, and moderation infrastructure.

### Security: Sandboxed Agent Contexts
Scraping/analysis agents are sandboxed (can only produce analysis text). Content creation agents generate posts. Publishing agents handle connectors. The Chief of Staff orchestrates but each sub-agent has limited permissions. Prevents prompt injection via competitor URLs.

## V1 Feature Scope

### Base Features (from Transition Plan T0→T6)

**Dashboard (Team Collaboration Layer):**
- Kanban task board with drag-and-drop status tracking
- Agent status cards — **Living agent cards**: online agents show amber pulse on status dot + live-updating task description ("Analyzing competitor X's latest post..."). Each card includes mini activity timeline (last 3 actions). Click card → expand to full run history. Feels like watching someone work, not reading a report.
- Approval queue — review agent-generated content (text + media preview), approve/reject/edit. **Edit = inline**: clicking Edit makes card text an editable textarea in-place. User edits, then "Approve" (saves + approves) or "Save Draft" (saves without approving). Media slots stay visible during edit.
- Run history with expandable detail (cost, duration, success/fail)
- Chat with agent directly in dashboard (REST polling V1)
- Team access — multiple users can view, comment, approve (all users are admins for V1)
- Action-First home page: pending approvals at top, agent status, intelligence brief below

**Content Pipeline (Agent Execution Layer):**
- Social media content generation flow: scrape trends → brainstorm → generate posts + images → surface for approval → schedule → publish
- Competitor monitoring: track competitor social accounts, surface insights
- Content calendar view: what's scheduled, what's published, what's pending

**Connectors (V1 Core — 5 connectors + framework):**

*Social platforms via unified API (evaluate Ayrshare vs Late vs PostForMe during T5a):*
1. X/Twitter — post (with images), monitor, scrape for content ideas
2. LinkedIn — post (with images), monitor company page

*Built separately (simple, well-documented APIs):*
3. Slack — notifications, approvals, chat with agent
4. Google Workspace (Docs/Sheets/Calendar) — content calendar, briefs
5. Email (Gmail/SMTP) — outreach, newsletters

**Connector Framework:**
- Social platforms: unified API handles OAuth tokens, posting, scheduling, analytics, rate limiting
- Slack/Google/Email: direct API integrations with shared auth refresh and error handling
- Connection status visible in dashboard
- Rate limit budget tracking per connector with auto-backoff
- All API routes return `{ error: string, code: string }` on failure

## First-Run Experience

When a new tenant signs up, the dashboard shows a **guided welcome flow** instead of empty states:

1. **Welcome screen**: "Welcome to PixelPort! Let's get your Chief of Staff running."
2. **Step 1**: Connect Slack (OAuth flow)
3. **Step 2**: Connect X and/or LinkedIn (via unified API OAuth)
4. **Step 3**: Add your first competitor URL (auto-discover social profiles)
5. **Step 4**: "Your Chief of Staff is starting its first scan..." → **Agent Activity Feed** shows live progress ("Scanning competitor website..." → "Analyzing your brand voice..." → "Drafting first batch of posts...")

Each step shows progress. The Activity Feed (expansion #6) serves double duty: onboarding progress + ongoing activity sidebar. Once first content is generated, the dashboard populates naturally and the welcome flow disappears.

## Interaction States

| Feature | Loading | Empty (post-onboarding) | Error | Success |
|---|---|---|---|---|
| Dashboard Home | Skeleton cards (amber shimmer pulse) | First-run welcome flow | "Agent workspace offline" + retry | Populated with live data |
| Approval Queue | Skeleton cards | "Inbox zero — your Chief handled everything" | Proxy error + retry | Cards render with media |
| Kanban Board | Column skeletons | "No tasks yet — Chief is working on it" + link to approvals | Proxy error + retry | Columns populate |
| Content Calendar | Grid skeleton | "Schedule your first post" + link to approvals | API error + retry | Events render |
| Chat | Message skeleton | "Say hello to your Chief of Staff" with suggested prompts | "Connection lost" + auto-reconnect | Messages render |
| Competitors | Card skeletons | "Add your first competitor" + URL input | Scrape failed + retry | Cards render |
| Brand Voice | Skeleton | "Calibrate your brand voice" + start button | Analysis failed + retry | Rules display |
| Agent Status | Pulse placeholder | "No agents provisioned" | "Droplet unreachable" + retry | Live status with amber pulse |

**Loading pattern:** Skeleton screens matching content shape (not spinners). Use `bg-muted` with amber shimmer animation.

**Error pattern:** Each error shows what went wrong + retry button. Auth errors include "Reconnect" link to /connections.

**Success feedback:** Toast notification on key actions — "Post approved and scheduled for Tue 3pm", "Published successfully to X"

## Responsive Design

**Mobile (< 768px):**
- Sidebar collapses to hamburger menu
- Approval queue: stacked cards, 44px+ touch targets for approve/reject/edit buttons
- Chat: full-width messages, input bar at bottom
- Other pages: "Best viewed on desktop" banner with link to mobile-friendly approvals

**Tablet (768-1024px):**
- Sidebar collapses, content fills width
- All pages functional, grid adjusts from 4→2 columns on stats

**Keyboard navigation:**
- Tab through interactive elements
- Enter/Space to approve/reject
- Escape to close modals/expanded cards
- Focus rings: `box-shadow: 0 0 0 3px rgba(212,168,83,0.1)` (amber, per DESIGN.md)

**Screen reader support:**
- ARIA labels on agent status dots ("Chief of Staff: online, drafting weekly brief")
- Approval action buttons labeled ("Approve post: Most startups don't fail...")
- Sidebar navigation landmarks
- Live region for toast notifications

## Error Handling (8 gaps fixed)

All error paths rescued with user-visible messages:
- Droplet unreachable → "Agent workspace offline" + retry
- OAuth expired → "Reconnect your X account" banner
- Content generation failed → "Regenerating..." + auto-retry
- Platform rejects content → Show rejection reason + edit option
- Connector disconnected → Block scheduling + "reconnect" CTA
- LLM refusal/malformed output → Retry with adjusted prompt

## Test Strategy (V1 — Lean)

Integration tests for critical paths only:
1. **Proxy auth** — `proxyToGateway()` forwards requests, handles auth failures, returns correct errors
2. **Publishing pipeline** — content flows from approval → Inngest schedule → unified API post → `external_post_id` stored
3. **Approval flow** — approve/reject/edit actions update content status correctly

Unit tests and E2E coverage expand as bugs surface post-launch.

## Success Criteria

1. A small B2B startup team can onboard, connect Slack + LinkedIn + X (via unified API), and see AI-generated social media content (with images) ready for approval within 24 hours
2. Team members can view agent activity, approve/reject content, and chat with the agent
3. At least one external team is using PixelPort weekly and willing to pay $200/month
4. Content pipeline produces posts that require <30% editing before comfortable publishing
5. Approval rate trends upward over 4 weeks (learning system working)
6. Weekly intelligence brief delivered every Monday (dashboard + Slack)
7. Connector framework allows adding a new connector in <4 hours of CC time
